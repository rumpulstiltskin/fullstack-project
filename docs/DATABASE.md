# Database Design

## Engine

SQLite, stored as a single file (`kanban.db`) in the backend working directory. Created automatically on first run if it does not exist.

## Schema

```sql
CREATE TABLE users (
    id       TEXT PRIMARY KEY,
    username TEXT UNIQUE NOT NULL,
    password TEXT NOT NULL
);

CREATE TABLE boards (
    id      TEXT PRIMARY KEY,
    user_id TEXT NOT NULL REFERENCES users(id),
    name    TEXT NOT NULL
);

CREATE TABLE columns (
    id       TEXT PRIMARY KEY,
    board_id TEXT NOT NULL REFERENCES boards(id),
    title    TEXT NOT NULL,
    position INTEGER NOT NULL
);

CREATE TABLE cards (
    id        TEXT PRIMARY KEY,
    column_id TEXT NOT NULL REFERENCES columns(id),
    title     TEXT NOT NULL,
    details   TEXT NOT NULL DEFAULT '',
    position  INTEGER NOT NULL
);
```

## Notes

- All IDs are random hex strings generated by the backend (matching the frontend's `createId` pattern).
- `position` is a zero-based integer used to order columns within a board and cards within a column.
- `password` is stored as plaintext for the MVP (the only user is hardcoded). It is a `NOT NULL` column so the schema is ready for real credentials later without migration.
- Foreign keys are declared but SQLite requires `PRAGMA foreign_keys = ON` per connection to enforce them — the backend will enable this.

## Seed data

On first run the backend will insert:

| Table   | Row |
|---------|-----|
| users   | id=`user-1`, username=`user`, password=`password` |
| boards  | id=`board-1`, user_id=`user-1`, name=`My Board` |
| columns | Five rows for Backlog / Discovery / In Progress / Review / Done, positions 0–4 |

No seed cards — the board starts empty.

## Future extensibility

- Multiple users: already supported by the `user_id` FK on `boards`.
- Multiple boards per user: already supported; the API will filter by the authenticated user.
- Password hashing: change the `password` column value in `users`; no schema migration needed.
